## 1. オンライン化の方針と範囲

本作は **シングルプレイ主体のハクスラRPG + オンライン連動機能** の構成を採る（原神 / Star Rail と同様）。リアルタイム協力プレイではなく、「クラウドセーブ」「グローバルランキング」「ワールドログ（他プレイヤーの発見を見る）」「第一発見者システム」を中核とする。

| カテゴリ | 具体機能 | 優先度 |
|---|---|---|
| **認証** | OAuth（Google / Discord）+ メール認証 | 🔴 最優先 |
| **クラウドセーブ** | ゲーム状態をサーバーに保存・デバイス間共有 | 🔴 最優先 |
| **グローバルランキング** | 残滓スコア / ステージクリアタイム / 累計ダメージ | 🟡 HIGH |
| **第一発見者システム** | UR/SSR アイテムの世界初討伐者刻印 | 🟡 HIGH |
| **ワールドログ** | UR発現・初討伐の世界通知フィード | 🟡 HIGH |
| **非同期PvP（将来）** | 防衛パーティを登録 → 他プレイヤーが挑戦 | 🟢 LOW |

## 2. 認証システム

### 2.1 技術選定: NextAuth.js v5

```
next-auth@5 (Auth.js) — Next.js 15 Server Actions と最も統合しやすい
```

対応プロバイダ:
- **Google OAuth** — 主要プロバイダ。モバイルブラウザでの UX が最良
- **Discord OAuth** — ゲーマー向けコミュニティとの連携
- **Email Magic Link** — プロバイダアカウントを持たないユーザー向け

### 2.2 セッション戦略

```typescript
// auth.ts (プロジェクトルート)
import NextAuth from 'next-auth';
import Google from 'next-auth/providers/google';
import Discord from 'next-auth/providers/discord';
import Resend from 'next-auth/providers/resend';
import { PrismaAdapter } from '@auth/prisma-adapter';
import { prisma } from '@/lib/prisma';

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: PrismaAdapter(prisma),
  providers: [Google, Discord, Resend({ from: 'noreply@necromancebrave.com' })],
  session: { strategy: 'database' },
  callbacks: {
    session: ({ session, user }) => ({
      ...session,
      user: { ...session.user, id: user.id },
    }),
  },
});
```

## 3. クラウドセーブ設計

### 3.1 保存トリガーとデータ分類

**常時サーバー保存（強整合性）**: ゲームの根幹に関わるデータ。

| イベント | 保存内容 | Server Action |
|---|---|---|
| ステージクリア | clearedStages, EXP, Gold, ドロップアイテム | `processStageResult()` |
| 転職 | currentJobId, UserJob(level/exp) | `changeJob()` |
| ネクロランクアップ | necroLevel, necroRank, necroMaxCost | `performRankUp()` |
| 装備変更 | equipment.weapon, equippedResidues | `updateEquipment()` |
| 魂石化 | SoulShard追加, Monster削除 | `createSoulShard()` |

**ローカル優先（結果整合性）**: 頻度が高く即時性が重要なデータ。ゲーム終了時や定期的にバッチで同期。

| データ | ローカル保存 | サーバー同期タイミング |
|---|---|---|
| battleLogs | Zustand のみ | 同期しない（UI表示用） |
| demonGauge | Zustand のみ | バトル終了時 |
| currentEnergy | Zustand のみ | バトル終了時 |

### 3.2 同期アーキテクチャ

```
クライアント                        サーバー (Next.js Server Actions)
──────────────────                  ────────────────────────────────
useGameStore (Zustand)
  │
  ├─ ステージクリア時
  │    └─ processStageResult()  ──→  GameManager.processStageResult()
  │                                    → Prisma $transaction
  │                                    → Item.create / Character.update
  │                             ←─── 更新後の CharacterData を返却
  │    └─ set({ player, inventoryItems, ... })
  │
  ├─ 装備変更時
  │    └─ updateEquipment()     ──→  GameManager.updateEquipment()
  │                             ←─── 更新後の equipment を返却
  │
  └─ 初回ロード
       └─ loadGameState()       ──→  GameManager.loadGameState(userId)
                                     → Character + Items + Residues を一括取得
                                ←─── GameState を返却
                                └─ initialize(gameState)
```

### 3.3 オフライン → オンライン の再接続戦略

オフライン中の操作はローカルで完結させ、オンライン復帰時にサーバーと照合する。

```typescript
// src/hooks/useOnlineSync.ts
export function useOnlineSync() {
  const pendingActions = useTempStore(s => s.pendingActions);

  useEffect(() => {
    const handleOnline = async () => {
      for (const action of pendingActions) {
        await syncAction(action);
      }
      clearPendingActions();
    };
    window.addEventListener('online', handleOnline);
    return () => window.removeEventListener('online', handleOnline);
  }, [pendingActions]);
}
```

コンフリクト解決ポリシー: **サーバー優先**。ローカルのみの変更（オフライン中のステージクリア）はサーバーへ提出し、サーバーが検証した結果を正として採用する。

## 4. バトル結果のサーバー検証（不正防止）

### 4.1 基本方針

クライアント側の `BattleEngine` は表示用の演出計算を担う。**ドロップ・EXP・ステージクリア判定はサーバーで独立して再計算する**。

```
クライアント:
  BattleEngine.simulateAction() → BattleLog[] → UI演出

バトル終了時:
  クライアント → processStageResult(stageId, turnCount, usedSkillIds)
  サーバー:
    [1] セッション検証（auth()）
    [2] stageId が解放済みかチェック
    [3] BattleEngine をサーバーで同一条件で再実行（シードベース乱数）
    [4] 結果が許容範囲内か検証
    [5] ドロップ抽選 → DB保存
```

### 4.2 シードベース乱数

```typescript
// BattleEngine.ts のコンストラクタに seed を追加

  this.rng = seedRandom(seed ?? Date.now());

// processStageResult の Server Action
```

## 5. グローバルランキング

### 5.1 ランキング種別

| ランキング名 | 集計軸 | リセット周期 |
|---|---|---|
| **残滓スコアボード** | `AbyssalResidue.residueScore` の最高値 | なし（永続） |
| **ステージクリアタイム** | `StageRecord.turnCount` の最少値（ステージ別） | なし（永続） |
| **累計ダメージ** | `CharacterStats.totalDamage` の累計 | 月次リセット |
| **魔王討伐数** | BOSSステージのクリア回数 | 月次リセット |

### 5.2 スキーマ追加

```prisma
// ランキング用スコアモデル
model StageRecord {
  id          String   @id @default(cuid())
  userId      String
  user        User     @relation(fields: [userId], references: [id])
  stageId     String
  turnCount   Int
  totalDamage Int
  clearedAt   DateTime @default(now())
  @@unique([userId, stageId])  // 1プレイヤー1ステージ1レコード（最良を保持）
}

model PlayerStats {
  userId         String   @id
  user           User     @relation(fields: [userId], references: [id])
  totalDamage    BigInt   @default(0)
  bossKillCount  Int      @default(0)
  bestResidueScore Float  @default(0)
  updatedAt      DateTime @updatedAt
}
```

### 5.3 ランキングのキャッシュ戦略

ランキングは全プレイヤーの集計が必要なため、毎リクエストのDB全件スキャンは不可。

```
Upstash Redis（Vercel 対応のサーバーレス Redis）でキャッシュ:
  - ランキングデータ: TTL 5分
  - 各プレイヤーのベストスコア更新時: キャッシュ invalidate

src/services/RankingService.ts:
  getRanking(type, limit=50) → Redis から取得（ミス時 PostgreSQL で再構築）
  updateScore(userId, type, value) → DB更新 + Redis の該当キャッシュを削除
```

### 5.4 ランキング API

```typescript

  const { searchParams } = new URL(req.url);
  const type = searchParams.get('type') ?? 'RESIDUE_SCORE';
```

## 6. 第一発見者システム（オンライン実装）

### 6.1 グローバル排他制御

複数プレイヤーが同時に同じアイテムを初めてドロップするとき、1人だけが「第一発見者」になる。

```typescript
// GameManager.ts — processStageResult() 内のドロップ処理
await prisma.$transaction(async (tx) => {
  const item = await tx.item.create({ data: { ...itemData } });

  // 第一発見者チェック（同じ itemId のレコードが存在しないか）
  const isFirst = !(await tx.item.findFirst({
    where: { name: item.name, isUnique: true, discovererId: { not: null } },
  }));

  if (isFirst && item.rarity === 'UR') {
    await tx.item.update({
      where: { id: item.id },
      data: {
        isUnique: true,
        discovererId: session.user.id,
        discovererName: session.user.displayName ?? session.user.name,
        serialNo: await getNextSerialNo(tx, item.name),
        discoveredAt: new Date(),
      },
    });
    // ワールドログへ通知
    await emitWorldEvent('UR_DISCOVERED', { userId: session.user.id, itemName: item.name });
  }
});
```

### 6.2 シリアルNo の採番

```prisma
// グローバルシーケンス管理
model ItemSerialCounter {
  itemName String @id
  counter  Int    @default(0)
}
```

```typescript
async function getNextSerialNo(tx: PrismaTx, itemName: string): Promise<number> {
  const result = await tx.itemSerialCounter.upsert({
    where: { itemName },
    update: { counter: { increment: 1 } },
    create: { itemName, counter: 1 },
  });
  return result.counter;
}
```

## 7. ワールドログ（リアルタイム通知）

### 7.1 技術選定: Pusher Channels（サーバーレス対応 WebSocket）

Vercel のサーバーレス環境では永続的 WebSocket は使えないため、**Pusher Channels**（マネージド WebSocket）を採用する。

```
イベント発生（サーバー）
  → Pusher SDK で world-log チャンネルに publish
  → クライアントが Pusher JS SDK で subscribe
  → リアルタイムでワールドログ UI に表示
```

### 7.2 ワールドイベント種別

| イベントID | 条件 | 表示文 |
|---|---|---|
| `UR_DISCOVERED` | UR アイテムを世界初ドロップ | 「⚡ {playerName} が世界初討伐！ [{itemName}] No.{serialNo}」 |
| `SSR_DISCOVERED` | SSR アイテムを世界初ドロップ | 「✦ {playerName} が [{itemName}] を初めて発見」 |
| `BOSS_CLEARED` | BOSS ステージクリア（世界初） | 「🏆 {playerName} が [{stageName}] を初めて攻略」 |
| `RANKING_UPDATED` | 残滓スコアランキング1位更新 | 「👑 {playerName} が残滓スコアランキング1位に」 |

### 7.3 実装

```typescript
// src/services/WorldEventService.ts
import Pusher from 'pusher';

const pusher = new Pusher({
  appId: process.env.PUSHER_APP_ID!,
  key: process.env.NEXT_PUBLIC_PUSHER_KEY!,
  secret: process.env.PUSHER_SECRET!,
  cluster: process.env.PUSHER_CLUSTER!,
});

export async function emitWorldEvent(
  eventType: WorldEventType,
  payload: Record<string, unknown>,
) {
  // DBにログとして保存
  await prisma.worldLog.create({
    data: { type: eventType, payload: JSON.stringify(payload), createdAt: new Date() },
  });
  // リアルタイム通知
  await pusher.trigger('world-log', eventType, payload);
}
```

```typescript
// クライアント側: src/hooks/useWorldLog.ts
import Pusher from 'pusher-js';
import { useEffect, useState } from 'react';

export function useWorldLog() {
  const [events, setEvents] = useState<WorldLogEntry[]>([]);

  useEffect(() => {
    const pusher = new Pusher(process.env.NEXT_PUBLIC_PUSHER_KEY!, {
      cluster: process.env.NEXT_PUBLIC_PUSHER_CLUSTER!,
    });
    const channel = pusher.subscribe('world-log');
    const handler = (data: WorldLogEntry) =>
      setEvents(prev => [data, ...prev].slice(0, 50));  // 最新50件

    Object.values(WORLD_EVENT_TYPES).forEach(t => channel.bind(t, handler));
    return () => pusher.unsubscribe('world-log');
  }, []);

  return events;
}
```

```prisma
// ワールドログ永続化
model WorldLog {
  id        String   @id @default(cuid())
  type      String   // WorldEventType
  payload   Json
  createdAt DateTime @default(now())
  @@index([createdAt])
}
```

### 11.4 DBマイグレーション安全運用ルール

1. `prisma migrate dev` はローカルのみ。本番は `prisma migrate deploy` を CI で実行
2. 破壊的マイグレーション（カラム削除・型変更）は **必ずメンテナンスウィンドウ**を設定
3. `@default` 付きの NOT NULL カラム追加は無停止デプロイ可
4. マイグレーション前後のバックアップは Neon / Supabase の自動バックアップに依存（手動スナップショット推奨）
