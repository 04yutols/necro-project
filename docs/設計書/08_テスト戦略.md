## 1. テスト方針

**モックは使わない（インテグレーションテストはリアルDBへ接続）**

設計意図: 過去にモックを使ったテストが全てパスしたにもかかわらず、本番でのDB migration 失敗を見逃した経緯があるため。

- **ユニットテスト**: 計算ロジック（BattleEngine / JobService / NecroService）を副作用なしで検証。モックデータはテスト内で定義するが、外部APIやDBはモックしない
- **インテグレーションテスト**: Prisma Client が本物のDB接続を持つ（テスト前に `beforeAll` でPrismaを初期化、`afterAll` で切断）
- **E2Eテスト**: Playwright が `http://localhost:3080` のdev serverに接続し、実際のブラウザ操作をシミュレート

### BattleEngine.test.ts — 検証詳細

```typescript
// FIRE_DMG_BOOST=50 のとき、炎スキルのダメージが属性加成なしより増えること
// ダメージ式: ATK × power × DEF軽減 × 属性加成 × 耐性 × 会心

test('applies element damage boosts for matching skill elements', () => {
  const baseEngine = new BattleEngine(mockPlayer, []);
  const boostedEngine = new BattleEngine({
    ...mockPlayer,
    elementDmgBoosts: { FIRE: 50 },
  }, []);
  const base = baseEngine.simulateAction('MAGIC_SKILL', mockTarget, 'skill_mage_1');
  const boosted = boostedEngine.simulateAction('MAGIC_SKILL', mockTarget, 'skill_mage_1');
  expect(boosted[0].damage).toBeGreaterThan(base[0].damage ?? 0);
});
```

**設計上の重要点**: `BattleEngine` は React / Zustand に依存しない純粋クラスのため、テストファイルで `new BattleEngine(player, monsters)` と直接インスタンス化できる。

## 5. テストカバレッジ方針

| 対象 | テスト種別 | 理由 |
|---|---|---|
| ダメージ計算式 | Jest ユニット | 数値が仕様通りかをコードレベルで保証 |
| RankUp / コスト検証 | Jest ユニット | エラーメッセージ文字列まで保証が必要 |
| パッシブ永続化 | Jest インテグレーション（実DB） | mock/prod 乖離を防ぐため実DBを使用 |
| バトルUI操作 | Playwright E2E | EN消費UIの連動はブラウザでしか検証できない |
| 魂石化UI | Playwright E2E | `window.confirm` ダイアログが絡むためE2Eが必要 |
| リザルト画面遷移 | Playwright E2E | アニメーション完了後の状態変化を検証 |

**カバレッジ対象外（意図的）**:
- PixiJS内部のレンダリング（VFX描画内容）— PixiJS は60fps描画を前提とし、テスト環境ではhttps://github.com/pixijs/pixijs
- Framer Motionのアニメーション途中状態 — CSSアニメーションはタイミング依存で不安定
- Zustand ストアの個別setter — ロジックはサービス層・エンジン層でカバー

## 6. BattleEngine をユニットテスト可能にした設計意図

`BattleEngine` は React / Zustand に依存しない純粋クラスとして実装されている。

```typescript
// src/logic/BattleEngine.ts
export class BattleEngine {
  constructor(player: CharacterData, monsters: (MonsterData | null)[]) { ... }
  simulateAction(type: string, target: any, skillId?: string): BattleLog[] { ... }
}
```

**意図**:
- `BattleCanvas.tsx` からは `actionTrigger` 経由で疎結合に呼び出す（直接インポートはしない）
- ユニットテストでは `new BattleEngine(player, monsters)` と直接インスタンス化し、計算結果を検証
- React/Next.js のセットアップなしで `npm test` が通る

## 7. テスト拡張候補

| 候補 | 理由 |
|---|---|
| `AbyssalResidue` 強化の成長式テスト | `newMaxExp = floor(maxExp × 1.5)` の数値精度保証 |
| WAVE進行テスト（10ターン毎にwave++） | `BattleEngine.updateState()` のカウンタロジック |
| 全体攻撃（ALL_ENEMIES）テスト | 実装後に BattleEngine 側を検証 |
| 属性無効（resistance=100）テスト | `elementMultiplier=0` で damage が最低保証1になること |
| 第一発見者システムのDB一意性テスト | `serialNo` のグローバルシーケンス採番が重複しないこと |
