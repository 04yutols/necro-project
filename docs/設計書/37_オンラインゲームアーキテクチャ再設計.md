## 1. 設計原則

### 1-1. データ分類の4軸

| 軸 | 問い |
|---|---|
| **永続性** | 永久保存が必要か / セッション中だけでよいか / デバイス単位でよいか |
| **アクセス頻度** | 毎リクエストで読むか / セッション開始時1回か / 低頻度か |
| **一貫性** | 複数書き込みの原子性が必要か / 最終整合性で許容できるか |
| **スコープ** | ユーザー単位か / グローバル（全ユーザー比較）か / デバイス単位か |

### 1-2. ツール適合マトリクス

| ツール | 得意なこと | 苦手なこと |
|---|---|---|
| **Neon/Prisma** | 永続・リレーショナル・ACID・複雑クエリ | 高頻度アクセス（コールドスタートあり）・ソートランキング |
| **Upstash Redis** | 高速読み取り・ソートセット・TTLキャッシュ・カウンタ | 複雑なリレーション・トランザクション |
| **Pusher** | リアルタイムブロードキャスト | データ保存・永続性 |
| **JWT Cookie** | リクエストごとの認証（DBなし） | セッション無効化（ログアウト後即時反映不可） |
| **Zustand (memory)** | バトル中の高頻度状態更新・セッションキャッシュ | ページリロード・デバイス間共有 |
| **localStorage (persist)** | デバイス設定・UX状態の記憶 | クロスデバイス同期・機密データ |
| **JSON files (master data)** | 読み取り専用ゲームバランスデータ | 動的変更・ユーザー固有データ |

## 2. データカタログ — どのデータをどこに置くか

### 2-1. Neon/Prisma（永続・正規化）

**対象データ**: ユーザー・キャラクター・ゲーム進行

| テーブル | データ | 書き込みタイミング |
|---|---|---|
| `User` | アカウント情報・displayName | 登録時・プロフィール変更時 |
| `Account` | OAuth連携情報 | OAuth接続時（NextAuth管理） |
| `Character` | ステータス・gold・necroExp・装備・パーティ構成・クリアステージ | ログイン時作成 / ステージクリア / 装備変更 / 転職 |
| `UserJob` | 職業ごとのlevel・exp | ステージクリア時（バトル結果） |
| `Monster` | 所持モンスター一覧 | ステージクリア時（ドロップ） |
| `Item` | 所持武器・消耗品一覧 | ステージクリア時（ドロップ） |
| `AbyssalResidue` | 所持残滓一覧・強化状態 | ステージクリア / 強化操作時 |
| `SoulShard` | 所持ソウルシャード | ステージクリア時 |
| `StageRecord` | クリア記録（ターン数・タイム・ダメージ） | ステージクリア時（ランキング用Source of Truth） |
| `PlayerStats` | 累計ダメージ・ボスキル数 | ステージクリア時 |
| `WorldLog` | 世界ログのアーカイブ | UR発見・ボス初討伐イベント発生時 |
| `ItemSerialCounter` | 第一発見者シリアル管理 | UR/Unique初ドロップ時 |

**Prismaを使わないもの**（重要）:
- セッション管理 → JWT Cookie で完結（`Session`テーブル不要）
- ランキング集計クエリのたびのDB直撃 → Redisキャッシュを経由する
- バトル中の状態 → クライアントZustandのみ
- マスターデータ（スキル・エネミー・ステージ） → JSONファイルをバンドル

---

### 2-2. Upstash Redis（高速・揮発）

**役割**: PostgreSQLの前段キャッシュ + ランキング専用ストレージ

| Redisキー | 型 | 内容 | TTL |
|---|---|---|---|
| `ranking:{stageId}:turn` | Sorted Set | スタージごとのターン数ランキング (score=turnCount) | 永続（要素削除のみ） |
| `ranking:{stageId}:time` | Sorted Set | クリアタイムランキング | 永続 |
| `ranking:player:damage` | Sorted Set | 累計ダメージランキング | 永続 |
| `ranking_cache:{type}` | JSON String | ランキング上位50件のキャッシュ | 300秒（5分） |
| `worldlog:recent` | List | 直近50件のワールドログ（新規接続時に初期表示） | 24時間 |

**Redisを使わないもの**:
- ユーザーのゲーム進行データ → Neon（永続性が必要）
- キャラクターデータのキャッシュ → セッション内はZustandで保持（Redis不要）
- セッション管理 → JWT（Redis Session不要）

---

### 2-3. Pusher Channels（リアルタイム配信）

**役割**: データ保存には使わない。WorldEventService がNeonに書いてから発火。

```
Server Action (WorldEventService)
  → prisma.worldLog.create()   … Neon に永続化
  → Redis LPUSH worldlog:recent … 新規接続用のスナップショット更新
  → Pusher trigger('world-log') … 現在接続中の全クライアントにブロードキャスト
```

**クライアントの初期化シーケンス**:
1. 接続時: `/api/worldlog/recent` から Redis の直近50件を取得（Neonは叩かない）
2. Pusherに subscribe → 以降はリアルタイム受信

---

### 2-4. JWT Cookie（認証）

**NextAuthのJWT戦略を維持する理由**:
- Server Actionが呼ばれるたびに `await auth()` でユーザーIDを取得できる
- DBセッションテーブルへの問い合わせが不要（Neonコールドスタートを避けられる）
- ゲームのような長時間セッションでは特に有効

```typescript
// 全Server Actionの冒頭で
const session = await auth();        // JWTを検証するだけ（DB不要）
if (!session?.user?.id) throw ...;   // 未認証は弾く
```

**JWT戦略の制約と許容**:
- ログアウト後もJWT有効期限内はトークンが有効（セッションDB invalidationができない）
- ゲームアプリとして許容範囲（バンク・医療ではないため）
- 有効期限を24時間に設定して実害を最小化

---

### 2-5. Zustand / インメモリ（セッションキャッシュ・バトル状態）

**ログイン後のデータフロー**:
```
loadCharacterAction()  ←  Neon (1回のみ)
    ↓
useGameStore（Zustand）  ←  セッション中はここから読む
    ↓
バトル開始  →  BattleEngine（純粋関数・DBなし）
              ↓
         バトル結果
              ↓
processStageResultAction()  →  Neon（バトルごとに1回）
```

**Zustandに置くもの（インメモリ・非永続）**:
- `player: CharacterData` — DBからロードしたキャラクターの作業コピー
- `party`, `inventoryMonsters`, `inventoryItems` — 同上
- `currentTab`, `isDemonMode`, `battleLogs` — UIランタイム状態
- `demonGauge`, `monsterCurrentHp` — バトル中の揮発状態

**Zustandに置かないもの**:
- バトル後に確定した進行データ → Server Actionで即座にNeonへ

---

### 2-6. localStorage / Zustand persist（デバイスローカル設定）

**対象**: クロスデバイス同期が不要なUX状態

| Store | 内容 | DB不要な理由 |
|---|---|---|
| `useTutorialStore` | チュートリアル完了フラグ・進行状態 | `clearedStages`からある程度復元可能。デバイス単位で十分 |
| `useStoryStore` | 閲覧済みシーンID・ストーリーフラグ | スキップできればよい。クロスデバイス同期の優先度低 |
| `useAudioStore` | BGM音量・SE音量・ミュート | デバイスごとの設定が自然 |

**将来の検討**: ユーザーから「別端末でもチュートリアルを飛ばしてほしい」という要望が出たら `clearedStages.length > 0` をトリガーにチュートリアルをスキップすればDB同期なしに解決できる。

---

### 2-7. JSONファイル / MasterDataService（マスターデータ）

**絶対にDBに入れないもの**:

```
src/data/master/
  jobs.json      — 職業定義・スキル解放テーブル
  enemies.json   — エネミースペック・ドロップ率
  stages.json    — ステージ構成・WAVEデータ
  items.json     — 武器マスター
  materials.json — 素材マスター
  demonForms.json — 魔神化フォームデータ
```

**理由**: ゲームバランスデータはコードとともにバージョン管理する。DBに入れると「どのバージョンのデータか」が追跡できなくなる。変更はコードデプロイで行う。

## 4. 認証・起動フロー

### 4-1. セッション状態の判定（`useAuthFlow` フック）

```
App起動
  │
  ├─ /api/auth/session フェッチ（JWT検証）
  │
  ├─ [セッションなし] → AuthGate（全画面ログイン/登録UI）
  │       ↓
  │   ログイン成功 → JWT Cookie 発行
  │       ↓
  │   loadCharacterAction() → CharacterなしならCharacterCreation画面
  │
  └─ [セッションあり]
          ↓
      loadCharacterAction()
          ↓
      [Characterなし] → CharacterCreation画面（初回専用）
          ↓
      [Characterあり] → useGameStore.loadFromServer(data)
          ↓
      HOMEタブ表示
```

### 4-2. `loadCharacterAction` の内部処理

```typescript
// 1回の Prisma クエリで必要なデータを全件取得
const char = await prisma.character.findFirst({
  where: { userId: session.user.id },
  include: {
    jobs:           true,   // UserJob[]
    equipWeapon:    true,   // 武器
    abyssalResidues: true,  // 残滓一覧
    // ※ Monster, Item は別クエリ（ownerId で絞り込み）
  },
});

// Monster と Item は Character と直接リレーションがないため別取得
const monsters = await prisma.monster.findMany({ where: { characterId: char.id } });
const items    = await prisma.item.findMany({ where: { ownerId: char.userId } });

// ServerGameData に変換して返す（Neon への接続はここで終わり）
```

### 4-3. データ同期タイミング一覧

| 操作 | 書き込み先 | タイミング | 方法 |
|---|---|---|---|
| ステージクリア | Neon（Character・UserJob・Item・Monster・StageRecord・PlayerStats） | バトル終了直後 | `processStageResultAction`（既実装） |
| UR/Unique発見 | Neon（WorldLog・ItemSerialCounter） + Redis（worldlog:recent） + Pusher | 同上 | `emitWorldEvent`（既実装） |
| ランキング更新 | Neon（StageRecord）→ Redis（Sorted Set） | 同上 | `RankingService.upsertRecord`（既実装） |
| 転職 | Neon（Character.currentJobId） | 転職ボタン確定時 | `saveGameStateAction` |
| 武器装備変更 | Neon（Character.equipWeaponId） | 装備確定時 | `saveGameStateAction` |
| パーティ編成 | Neon（Character.partySlot0〜2Id） | 編成確定時 | `saveGameStateAction` |
| 残滓装備 | Neon（Character.equippedResidue0〜2Id） | スロットセット時 | `saveGameStateAction` |
| ログアウト | なし（JWT破棄のみ） | ログアウトボタン | NextAuth signOut |

**バトル中はNeonに一切書かない**。バトルが異常終了した場合は進行が保存されないが、これはゲーム設計として許容する（ステージはやり直し可能）。

## 6. 実装しないもの・判断理由

| 候補 | 判断 | 理由 |
|---|---|---|
| Redis にキャラクターデータをキャッシュ | ❌ 不採用 | セッション開始時の1回しか読まないのでZustandで十分。Redisの追加コストに見合わない |
| DBセッションテーブル（NextAuth strategy:'database'） | ❌ 不採用 | JWTで十分。Neonコールドスタートを毎リクエスト叩くのを避ける |
| バトル中の中間状態をDBに保存 | ❌ 不採用 | 複雑さに対してリターンが薄い。ステージやり直しを許容する設計 |
| チュートリアル・ストーリーフラグのDB同期 | ❌ 不採用（V1） | `clearedStages` でスキップ判定できる。要望があればV2で追加 |
| ランキングをNeon直接クエリ | ❌ 不採用 | 全ユーザーのソート集計はNeonには重い。Redis Sorted Setに委ねる |
| WebSocketサーバー自前運用 | ❌ 不採用 | Vercelサーバーレスと相性が悪い。Pusherに委ねる |
