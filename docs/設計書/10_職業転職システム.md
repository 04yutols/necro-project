# 10 — 職業・転職システム詳細設計

> 最終更新: 2026-05-06  
> 対象実装: `src/data/master/jobs.json`, `src/data/master/skills.json`, `src/logic/JobSystem.ts`, `src/store/useGameStore.ts`, `src/components/job/JobChangeScreen.tsx`

## 1. 目的

職業システムは「勇者の力」をプレイヤー自身の成長軸として扱う。
死霊術が軍団を強くする横軸であるのに対し、職業は主人公のステータス、Energy設計、使用スキル、戦闘テンポを変える縦軸になる。

転職UIの主目的は以下。

| 目的 | 内容 |
|---|---|
| 転職体験 | 現在職と候補職の見た目・役割・解放条件を一目で理解できる |
| ステータス変化 | 転職後の8ステータス差分を即時プレビューする |
| スキル変化 | 職業レベルに応じて使えるスキルと未解放スキルを確認できる |
| 将来拡張 | DB永続化、職業専用VFX、Tier3職業へ拡張できる |

## 2. 職業データ構造

職業マスターは `src/data/master/jobs.json` に集約する。

```ts
interface JobData {
  name: string;
  displayName?: string;
  nameEn?: string;
  title?: string;
  tier: number;
  category: 'PHYSICAL' | 'MAGICAL';
  role?: string;
  description?: string;
  unlock?: {
    jobs?: { jobId: string; minLevel: number }[];
    clearedStageId?: string;
  };
  statModifiers?: Partial<BaseStats>;
  energyCurve: {
    baseMaxEnergy: number;
    energyRegen: number;
    ultimateCost: number;
  };
  levelBonuses: Record<string, Partial<PassiveBonuses>>;
  skills: { level: number; skillId: string }[];
}
```

`category` は職業UXと習得スキル傾向の分類に使う。ダメージ計算の主軸は新ステータス設計に合わせて `ATK/DEF` に統一し、スキル単位の `element` と `attackType` で属性・演出差を表現する。

## 3. ステータス補正

プレイヤーは職業非依存の `baseStats` を持つ。
転職時は `baseStats x selectedJob.statModifiers` で現在ステータスを再計算する。

```text
jobStats[key] = round(baseStats[key] * statModifiers[key])
```

永続パッシブは `player.passives` に蓄積し、UIプレビューでは `jobStats + passives` を総合値として表示する。
BattleEngine は既存通り `getTotalStats()` でパッシブを加算するため、`player.stats` 自体にはパッシブを混ぜない。

## 4. 転職条件

Tier1は初期解放。
Tier2は2つの下位職レベルを条件にする。

| 職業 | 解放条件 |
|---|---|
| 魔剣士 | 剣士 Lv20 + 闇術師 Lv10 |
| ベルセルク | 剣士 Lv20 + 盗賊 Lv10 |
| アークメイジ | 魔法使い Lv20 + 闇術師 Lv10 |
| ソーサラー | 魔法使い Lv20 + 盗賊 Lv10 |
| ウォーロック | 闇術師 Lv20 + 魔法使い Lv10 |
| ネクロマンサー | 闇術師 Lv20 + 剣士 Lv10 |
| アサシン | 盗賊 Lv20 + 剣士 Lv10 |
| トリックスター | 盗賊 Lv20 + 魔法使い Lv10 |

条件判定は `getJobUnlockStatus(character, job)` に集約する。

## 5. スキル解決

職業スキルは `jobs.json.skills[]` から解決する。

```text
currentJobId
  -> jobs.json[currentJobId]
  -> player.jobs[currentJobId].level
  -> level >= skill.level の skillId を抽出
  -> skills.json[skillId] を BattleCanvas のスキルボタンへ変換
```

このため、転職するとバトル中の「術」メニューも現在職業のスキルへ切り替わる。
魔神化中だけは職業スキルより魔神技を優先する。

## 6. UI設計

画面名: `JobChangeScreen`

Genshin / Star Rail 的なキャラクター育成UXを参考に、以下の情報密度で構成する。

| 領域 | 内容 |
|---|---|
| Header | 影の儀式場、戻る、現在職業の小シジル |
| Hero | 選択職業の大型シジル、Tier、カテゴリ、称号、説明 |
| Job Rail | 職業一覧、Lv、LOCK状態を横スクロールで選択 |
| Stat Preview | 9ステータスの転職後値と現在職との差分 |
| Skill Preview | 習得済み/未習得スキル、属性、攻撃種別 |
| Footer Action | 戻る / 転職 / 条件未達成 |

iOS Safari 対策として、画面ルートの `motion.div` と `overflow` を分離する。
全体は `100dvh` の SPA フレーム内でスクロールし、ボタンは `env(safe-area-inset-bottom)` を考慮する。

### iOS レスポンシブ制約

実機Safariでは、横スクロールコンテナやGrid内の子要素に `min-width: auto` が残ると画面全体が横へ広がることがある。
転職画面では以下を必須ルールにする。

| 対象 | ルール |
|---|---|
| 画面ルート | `width: 100%`, `max-width: 100vw`, `overflow-x: hidden` |
| スクロール本文 | `min-width: 0`, `overflow-x: hidden`, `-webkit-overflow-scrolling: touch` |
| 横スクロール職業Rail | 親を `overflow: hidden`、Railだけ `overflow-x: auto` |
| ステータスGrid | `repeat(auto-fit, minmax(88px, 1fr))` を基本にし、430px以下は3列、360px以下は2列へ固定 |
| カード/ボタン | `box-sizing: border-box`, `min-width: 0` を指定 |
| 長い職業名/称号 | `overflow-wrap: anywhere` で折り返す |

実装では `job-change-screen` 系クラスを画面専用のレスポンシブ境界として扱う。
横スクロールが必要なのは職業Railだけに限定し、Railのスクロールバーは非表示にする。
大型シジルはiPhone幅でHero領域を圧迫しないよう、標準サイズを140pxに抑える。
