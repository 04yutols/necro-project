> 作成日: 2026-05-17

## 1. なぜログインフローが重要か

ゲームのすべての永続データはDBの `Character` レコードを起点として連鎖している。
ログイン→キャラクター作成の流れが正しく設計されないと、
**進行度やアイテムが消える・別アカウントのデータが混ざる・キャラクター未作成のままゲームが始まる** 事故が起きる。

```
User (認証)
 └─ Character (ゲームの中心)
       ├─ UserJob[]          職業レベル・EXP
       ├─ Monster[]          所持モンスター
       ├─ AbyssalResidue[]   所持残滓
       ├─ equipWeaponId      装備中の武器
       ├─ partySlot0〜2Id    出撃パーティ
       ├─ equippedResidue0〜4Id  装備残滓スロット
       ├─ gold / necroExp    資産・成長
       └─ clearedStages[]    進行フラグ

User (認証) ←─ Item[]        所持武器・消耗品 (ownerId → User.id)
User (認証) ←─ StageRecord[] クリア履歴・ランキング
User (認証) ←─ PlayerStats   累計ダメージ等
```

> **Item だけ User に紐づく理由**: 第一発見者システムで `discovererName` を User 単位で管理するため。
> ゲームプレイ上は Character と 1:1 なので実害なし（V1）。

## 3. 新規登録フロー（詳細）

### Step 1 — AuthGate 表示

- ゲームコンテンツは一切表示しない（真っ暗背景 + ロゴ + フォーム）
- タブ「ログイン」「新規登録」を並列表示
- 「新規登録」タブを選択

### Step 2 — 入力フォーム

| フィールド | バリデーション | DB格納先 |
|---|---|---|
| メールアドレス | RFC5322形式・重複チェック | `User.email` |
| パスワード | 8文字以上 | `User.passwordHash`（bcrypt 12rounds） |
| プレイヤー名 | 2〜16文字 | `User.displayName` |

### Step 3 — 登録 API

```
POST /api/auth/signup
  body: { email, password, displayName }
  
  1. email 重複チェック（prisma.user.findUnique）
  2. password バリデーション（8文字以上）
  3. displayName バリデーション（2〜16文字）
  4. bcrypt.hash(password, 12)
  5. prisma.user.create({ email, passwordHash, displayName, name: displayName })
  
  Response 200: { success: true }
  Response 400: { success: false, error: "このメールアドレスは既に登録されています" }
```

### Step 4 — 自動ログイン

登録成功後、画面遷移なしに即座にログイン処理を実行する。

```
POST /api/auth/signin/credentials
  body: { csrfToken, email, password }
  
  → NextAuth が authorize() を呼ぶ
  → JWT Cookie 発行（有効期限: 24時間）
```

### Step 5 — キャラクター作成画面へ遷移

`loadCharacterAction()` → null（新規ユーザーには Character がない）

`CharacterCreation` 画面を表示:
- 画面全体をゲームの世界観で演出（プロローグ風）
- 職業選択: warrior / mage / dark_priest / rogue の4択
- キャラクター名: `User.displayName` を初期値として入力欄に表示（変更可）
- 「冒険開始」ボタン

### Step 6 — キャラクター作成 Server Action

```typescript
// createCharacterAction(jobId: string, characterName: string)
'use server';

1. const session = await auth();  // JWT検証
2. セッションなし → Error('Unauthorized')

3. 既存 Character チェック
   if (prisma.character.findFirst({ where: { userId } })) → Error('既にキャラクターが存在します')

4. マスターデータから初期ステータスを取得
   const job = MasterDataService.getInstance().getJob(jobId);
   const baseStats = calculateInitialStats(job);

5. prisma.$transaction([
   // Character 作成
   prisma.character.create({
     data: {
       userId,
       name: characterName,
       currentJobId: jobId,
       ...baseStats,          // hp, atk, def, spd, critRate, critDmg, effectHit, effectRes
       gold: 50000,           // 初期所持金
       necroExp: 0,
       necroLevel: 1,
       necroRank: 1,
       necroMaxCost: 10,
       necroBaseStatsBonus: 1.0,
       clearedStages: [],
     }
   }),
   // 開始職業の UserJob 作成
   prisma.userJob.create({
     data: { characterId, jobId, level: 1, exp: 0 }
   }),
   // 初期装備武器の作成（マスターデータの R 武器）
   prisma.item.create({
     data: { ...STARTER_WEAPON, ownerId: userId }
   }),
])

6. loadCharacterAction() で全データを取得して返す
```

### Step 7 — ゲーム開始

```typescript
const data = await createCharacterAction(jobId, characterName);
useGameStore.getState().loadFromServer(data);
// → HOMEタブへ遷移
```

## 4. ログインフロー（詳細）

### Step 1 — AuthGate 表示（「ログイン」タブ）

### Step 2 — 入力フォーム

| フィールド | バリデーション |
|---|---|
| メールアドレス | 入力必須 |
| パスワード | 入力必須（8文字以上） |

### Step 3 — ログイン API

```
POST /api/auth/signin/credentials
  body: { csrfToken, email, password, redirect: 'false' }
  
  NextAuth.authorize():
    1. prisma.user.findUnique({ where: { email } })
    2. User がない → null（認証失敗）
    3. bcrypt.compare(password, user.passwordHash)
    4. 不一致 → null（認証失敗）
    5. 一致 → { id, name, email } を返す
    6. JWT Cookie 発行
  
  失敗レスポンス: { error: "CredentialsSignin" }
  → UI表示: "メールアドレスまたはパスワードが違います"
```

### Step 4 — キャラクターロード

```typescript
// loadCharacterAction()
'use server';

const session = await auth();
const char = await prisma.character.findFirst({
  where: { userId: session.user.id },
  include: {
    jobs: true,
    equipWeapon: true,
    equipSub: true,
    // ... 全装備スロット
    inventoryMonsters: {
      include: { soulShard: true, spiritCore: true }
    },
    abyssalResidues: true,
    equippedResidue0: true,
    equippedResidue1: true,
    // ... 全残滓スロット
    partySlot0: true,
    partySlot1: true,
    partySlot2: true,
  }
});

// Item (武器・消耗品) は User に紐づくため別クエリ
const items = await prisma.item.findMany({
  where: { ownerId: session.user.id }
});

// ServerGameData に変換して返す
return convertToServerGameData(char, items);
```

### Step 5 — Zustand に反映してゲーム開始

```typescript
useGameStore.getState().loadFromServer(data);
// → HOMEタブへ遷移
```

## 5. データバインディング — 登録完了時に何が作られるか

```
新規登録完了時のDB状態:

User
  id: "uuid-xxxx"
  email: "user@example.com"
  passwordHash: "$2b$12$..."
  displayName: "アルド"

Character
  id: "uuid-yyyy"
  userId: "uuid-xxxx"        ← User に紐づく
  name: "アルド"             ← 初期値は displayName（変更可）
  currentJobId: "warrior"
  hp: 7200, atk: 1250, ...   ← マスターデータから計算
  gold: 50000
  necroLevel: 1, necroExp: 0
  clearedStages: []

UserJob
  characterId: "uuid-yyyy"   ← Character に紐づく
  jobId: "warrior"
  level: 1, exp: 0

Item (初期武器)
  id: "uuid-zzzz"
  ownerId: "uuid-xxxx"       ← User に紐づく（武器の所有）
  name: "骨砕きの短剣"
  type: "WEAPON"
  rarity: "R"
  ...

```

```
ステージクリアごと:
  UserJob.exp/level が更新
```

## 6. セッション管理

### JWT Cookie の仕様

| 項目 | 値 | 理由 |
|---|---|---|
| 有効期限 | 24時間 | ゲームセッションとして十分。長すぎると乗っ取りリスク |
| 保存先 | httpOnly Cookie | XSS対策。JavaScriptからアクセス不可 |
| secure | true（本番） | HTTPS 必須 |
| sameSite | lax | CSRF 対策と通常遷移の両立 |

### セッション確認の実装

```typescript
// /api/auth/session が返すのは NextAuth のセッション情報
// JWT 検証はサーバー側で完結（DB なし）

// クライアント側での確認
const response = await fetch('/api/auth/session', { cache: 'no-store' });
const session = await response.json();
// session.user があれば認証済み
// session.user がなければ未認証（{} を返す）
```

### リロード時の動作

```
ページリロード
  │
  ├─ Zustand (memory) がクリア → player = null になる
  │
  └─ useAuthFlow が起動
       → /api/auth/session で Cookie 確認
       → Cookie 有効 → loadCharacterAction() でDB再取得
       → Zustand 再セット → ゲーム復元
```

## 7. セッション切れ（ゲーム中の対応）

### 発生タイミング

- JWT Cookie が 24時間後に期限切れ
- Server Action が `session = null` を返す

### 検出方法

```typescript
// Server Action でのチェック
const session = await auth();
if (!session?.user?.id) {
  return { error: 'SESSION_EXPIRED' };
}
```

### クライアント側の対応

```typescript
// processStageResultAction 等の戻り値を確認
const result = await processStageResultAction(...);
if (result?.error === 'SESSION_EXPIRED') {
  // Zustand の player データは保持したまま
  // 小さなリログインモーダルを表示
  showReloginModal();
}
```

### リログインモーダルの仕様

- ゲーム画面の上に半透明オーバーレイ
- メッセージ: 「セッションの有効期限が切れました。再度ログインしてください。」
- フォーム: メールアドレス・パスワード入力のみ（簡易版）
- 再ログイン成功 → モーダルを閉じてゲーム継続（Zustandデータは消えていないため）
- ゲームプレイを中断しない設計

## 8. ログアウトフロー

```
「ログアウト」ボタン押下
  │
  ├─ 確認ダイアログ: 「ゲームデータはクラウドに保存されています。」
  │
  └─ 確認
      │
      ├─ POST /api/auth/signout
      │   → NextAuth が JWT Cookie を削除
      │
      ├─ useGameStore.getState().clearServerData()
      │   → player, inventoryMonsters, inventoryItems... を全クリア
      │
      └─ AuthGate を全画面表示
```

> **ローカルストアはクリアしない**: `useTutorialStore`, `useStoryStore`, `useAudioStore` は
> ログアウトしても localStorage に残す。次回ログイン時も設定が引き継がれる方が UX が良い。

## 9. エラーハンドリング一覧

### 新規登録

| エラー | 原因 | UI表示 |
|---|---|---|
| メールアドレス重複 | 既登録のメール | 「このメールアドレスは既に登録されています」 |
| パスワード短すぎ | 8文字未満 | 「パスワードは8文字以上にしてください」 |
| プレイヤー名不正 | 2文字未満/16文字超 | 「プレイヤー名は2〜16文字にしてください」 |
| サーバーエラー | DB障害など | 「登録に失敗しました。しばらく経ってから再試行してください」|
| 登録後ログイン失敗 | CSRF失効など | 「ログインページからログインしてください」（稀なケース） |

### ログイン

| エラー | 原因 | UI表示 |
|---|---|---|
| 認証失敗 | メール未登録 / パスワード不一致 | 「メールアドレスまたはパスワードが違います」（詳細は隠す） |
| API接続不可 | サーバー停止 | 「サーバーに接続できません。しばらく経ってから再試行してください」 |

### キャラクター作成

| エラー | 原因 | UI表示 |
|---|---|---|
| 作成失敗 | DB障害など | 「キャラクターの作成に失敗しました。再試行してください」（リトライボタン表示） |
| 重複作成試行 | 二重タップなど | 冪等チェックで弾く（既存Character返す） |

### キャラクターロード

| エラー | 原因 | 対応 |
|---|---|---|
| Character なし（ログイン後） | データ不整合（稀） | CharacterCreation 画面を表示 |
| Neon 接続タイムアウト | コールドスタート中 | リトライ（最大3回、1秒間隔）→ 失敗でエラー画面 |

## 12. 実装フェーズ（最小起動までの順序）

```
1. AuthGate.tsx 作成 [完了]
   └─ 現行 AuthPanel のフォームロジックを全画面UIに移植

2. loadCharacterAction.ts 作成 [完了]
   └─ Prisma で Character + 全リレーションを1クエリで取得
   └─ ServerGameData 型を定義して返す

3. useGameStore に loadFromServer / clearServerData を追加 [完了]
   └─ ServerGameData → Zustand の全フィールドに展開

4. useAuthFlow フック作成 [完了]
   └─ セッション確認 → Character確認 → store反映 の状態管理

5. page.tsx 修正 [完了]
   └─ if (!player) initialize() を削除
   └─ <GameShell> に useAuthFlow を組み込む
   └─ state ごとに AuthGate / CharacterCreation / LoadingScreen / GameContent を振り分け

6. createCharacterAction.ts 作成 [完了]
   └─ Character + UserJob + 初期 Item を $transaction で作成

7. CharacterCreation.tsx 作成 [完了]
   └─ 職業カード選択UI → createCharacterAction 呼び出し
```

### 12.1 実装メモ（2026-05-17）

- Server Action は既存構成に合わせて `src/app/actions.ts` に集約した。
- `useAuthFlow` は `CHECKING / AUTH_GATE / LOADING_CHARACTER / CHARACTER_CREATION / READY / SESSION_EXPIRED` 相当の状態を持つ。
- Neon のコールドスタート対策として `loadCharacterAction()` は最大3回、1秒間隔で再試行する。
- `SESSION_EXPIRED` は Zustand のゲームデータを消さず、`ReloginModal` をゲーム画面上に表示する。
- 新規登録後は Credentials ログインを即時実行し、Character がない場合のみ `CharacterCreation` へ遷移する。
- `CharacterCreation` のキャラクター名初期値は `User.displayName`、未取得時はメールローカル部、最後に `アルド` を使う。
- JWT セッション有効期限は `src/auth.ts` で 24時間に設定した。
