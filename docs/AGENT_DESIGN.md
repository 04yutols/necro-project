Gemini CLI マルチエージェント・開発シリアルワークフロー設計書
1. コンセプト
本設計は、Claude CodeのAgentTeamのような協調動作をGemini CLIのSubAgent機能で再現し、各工程で「物理的な中間成果物」を残しながら直列（シリアル）に進行する。また、人間が睡眠中に記憶を整理するように、定期的な「情報のクリーンアップと統合」を行うフェーズを設けることで、長期的なプロジェクトの整合性を維持する。

2. ワークフロー全体図
Planner → Architect → Developer → Reviewer → QA Engineer → （任意タイミング） → Librarian

3. エージェント定義と責務
各エージェントは .gemini/agents/ 配下に定義し、特定の役割と制約を持たせる。

① Planner（タスク作成者）
ファイル名: planner.md

責務: ユーザーの曖昧な依頼を解釈し、実行可能な「やることリスト」に分解する。

成果物: docs/01_plan.md

制約: 常に「何を作るか」の境界線を明確に定義すること。

② Architect（設計者）
ファイル名: architect.md

責務: 計画に基づき、技術スタック、ディレクトリ構成、主要なインターフェース（API定義等）を決定する。

成果物: docs/02_architecture.md

制約: 実装者が迷わないよう、具体的なファイルパスとデータ構造を明記すること。

③ Developer（実装者）
ファイル名: developer.md

責務: 設計書に従い、コードを生成・編集する。

成果物: ソースコード（src/ 等）

ツール: read_file, write_file, make_directory

制約: 中略を禁止し、常に完全なソースコードを出力すること。

④ Reviewer（レビュアー）
ファイル名: reviewer.md

責務: 実装されたコードと設計の乖離、バグ、保守性をチェックする。

成果物: docs/04_review_report.md

制約: 指摘事項がある場合は具体的な修正案を提示し、合格時のみ「Approved」を出すこと。

⑤ QA Engineer（品質保証）
ファイル名: qa_engineer.md

責務: 正常系・異常系のテストケース作成、および実際の動作検証。

成果物: docs/05_qa_report.md

ツール: code_execution

制約: 実際にコードを実行し、出力結果に基づいたエビデンスを残すこと。

4. 記憶の整理フェーズ：Librarian（司書）
人間が睡眠をとるように、作業ログを「長期記憶」へと統合する特別なフェーズ。

ファイル名: librarian.md

責務: docs/ 内の各成果物を読み込み、ノイズ（修正された古い計画など）を排除して、プロジェクトの「正史」を編纂する。

成果物: PROJECT_WIKI.md または ARCHIVE/ へのログ退避

実行タイミング:

大きな機能（マイルストーン）の実装完了時。

コンテキストが肥大化し、AIの応答精度が低下し始めた時。

その日の作業の締めくくり。

5. アンチ・サボり・プロトコル（品質管理）
AI特有の「中略・省略」を物理的に封じ込めるための共通制約。

全行上書きの原則: ツール（write_file）を使用する際は、変更箇所のみの提示ではなく、必ずファイル全体の完全な内容を書き出させる。

検閲機能: Reviewer および Librarian の指示に「... や 中略、(以下略) が含まれていた場合、即座に Reject し再生成を要求する」というロジックを組み込む。

証跡の物理化: AIの回答文（チャットログ）だけでなく、必ず docs/ 配下のマークダウンファイルとして出力を強制し、Git等のバージョン管理に乗せる。

6. 運用方法（司令塔プロンプト例）
